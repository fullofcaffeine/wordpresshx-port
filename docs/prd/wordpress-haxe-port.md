## 0. Execution directive for Codex and other agents

This document is the initial product and architecture authority for the experiment. Beads is the execution ledger. Git is the source-history authority. Manifests and receipts are the evidence layer.

Agents must follow these rules:

1. **Do not begin broad source translation before the feasibility gates in §15 pass.** The first work is source locking, inventories, oracle infrastructure, and narrow compiler/backend spikes.
2. **Behavioral parity is the authority.** Upstream WordPress and Gutenberg are the behavioral oracles. A cleaner design is not a valid reason to change observable behavior before parity.
3. **Haxe is the only hand-authored implementation language for migrated runtime logic.** Generated PHP, TypeScript, TSX, and JavaScript are build artifacts. Temporary non-Haxe bootstrap tooling must be tracked as migration debt.
4. **Never edit generated target files by hand.** Fix Haxe source, a macro, the ABI linker, Genes, or the Haxe PHP backend.
5. **Never hide unsupported constructs.** Every unsupported construct becomes a machine-readable compiler-pressure item with an upstream fixture, minimized reproduction, owner, fallback, and removal condition.
6. **Never weaken or delete an upstream test merely to make the port pass.** Normalize nondeterministic output explicitly and version the normalization.
7. **Every original executable source file and every shipped executable artifact must have a manifest record.** There are no silent exclusions.
8. **Use Beads exclusively for task state.** Do not maintain parallel Markdown TODO lists, ad hoc issue files, or a second task database. The logical keys in this PRD, such as `WPHX-102`, are stable external references; actual Beads issue IDs are generated by `bd`.
9. **Start every work session with Beads context.** Run `bd prime`, then `bd ready --json`, inspect a candidate with `bd show <id> --json`, and atomically claim it with `bd update <id> --claim` before editing owned paths.
10. **Record discovered work immediately.** Create a Beads issue and link it with `discovered-from`; do not silently expand the current task. Use `bd remember` for durable project insights that are not actionable tasks.
11. **Use small vertical slices.** A slice is done only when it builds, runs, passes its oracle tests, updates coverage manifests, and emits a verification receipt.
12. **Close only verified work.** Before `bd close`, acceptance commands must pass from a clean worktree, the receipt must exist, generated output must be reproducible, and required manifest updates must be committed.
13. **Escalate target customization in order:** ordinary Haxe → target-specific Haxe APIs and macros → deterministic output linker → minimal Haxe PHP backend fork. Do not fork the compiler for formatting preferences.
14. **Keep Codex instructions layered and practical.** Root `AGENTS.md` contains repository-wide rules; target- or directory-specific rules belong in nested `AGENTS.md` files. The PRD stays in `docs/prd/` rather than being copied into agent instructions.
15. **Never hand-edit Beads storage.** Use `bd` commands. Treat `.beads/issues.jsonl` as a review/interchange export only, and use Dolt-native sync and `bd backup` for recovery.

The first Codex session must create the repository structure in §9, initialize Beads and the Codex integration in §18, create the bootstrap epics/tasks from §15, and execute Gate G0 followed by feasibility spikes F1–F10. It must not mass-convert WordPress or Gutenberg files.

## 1. Executive decision

### 1.1 Recommendation

Proceed with the experiment. The project is technically plausible, but it is not primarily a syntax-conversion problem. It is a **runtime and ecosystem compatibility program** whose hardest part is preserving WordPress’s procedural PHP contract:

- thousands of global functions;
- exact function signatures, default values, variadics, and references;
- global variables and superglobals;
- native PHP array behavior;
- conditional and pluggable declarations;
- include order and include-time side effects;
- mixed PHP/HTML templates and caller-scope behavior;
- reflection, serialization, callbacks, hooks, headers, buffering, and error behavior;
- exact public classes and file paths expected by plugins and themes.

The recommended architecture is therefore:

> **Haxe-authored implementations + generated WordPress ABI façades + generated original-path PHP shells + deterministic distribution linker.**

For browser code:

> **Haxe-authored packages + genes-ts TypeScript/TSX output + the upstream-compatible bundling pipeline**, with classic Genes JavaScript retained as a differential fallback until the TypeScript path is proven.

### 1.2 Direct answer: can macros adjust the Haxe PHP output?

**Yes, but macros alone are not the right complete solution.**

Haxe macros can:

- inspect and transform Haxe syntax before typing;
- generate Haxe modules, classes, wrappers, externs, and metadata;
- inspect typed modules;
- emit ABI manifests and companion files;
- use `php.Syntax.code(...)` for narrow target-language constructs;
- schedule post-generation work with `Context.onAfterGenerate`;
- attach target metadata such as native names and PHP-specific annotations.

Macros cannot cleanly turn the stock PHP target into a general top-level WordPress script emitter. The stock target is centered on namespaced, file-per-type classes and a generated bootstrap. WordPress requires global functions, include-time declaration boundaries, mixed templates, and original file topology. `onAfterGenerate` runs after normal generation; it can create or rewrite files, but a rewrite-heavy macro would be difficult to reason about and fragile across compiler changes.

Use macros to produce a typed **emission manifest**, then let a deterministic `wphx-link` stage generate the exact WordPress-facing files. Fork `genphp7.ml` only for semantics that cannot be implemented faithfully or efficiently through target APIs plus the linker.

### 1.3 Portability decision

The proposed **target-shaped Haxe first, portability later** stance is correct.

Do not insert a broad cross-target abstraction layer during the parity phase. It would increase code, runtime indirection, agent context, and semantic drift. Instead, mark host seams explicitly and keep internal code typed. After parity, extract abstractions only where repeated evidence shows a stable cross-target concept.

Use **seam markers, not speculative abstractions**:

- `@:wp.phpNative`
- `@:wp.jsNative`
- `@:wp.global`
- `@:wp.globalVar`
- `@:wp.template`
- `@:wp.hookBoundary`
- `@:wp.serializationBoundary`
- `@:wp.portabilityCandidate`

This preserves future options without burdening the initial implementation.

### 1.4 Codex + Beads execution decision

Use **vanilla Codex + Beads as the complete initial control plane**.

- Codex inspects repositories, edits source, runs commands, creates tests, reviews diffs, and executes the task packet.
- `AGENTS.md` supplies durable repository and directory-specific operating rules to Codex.
- Beads owns issues, dependencies, priorities, claims, progress, discovered work, and durable agent memory.
- Git and worktrees own source isolation, history, review, and integration.
- `wphx` manifests and verification receipts own coverage and proof.
- Generated dashboards are read models; they never become another task store.

CAF is not part of the bootstrap path. No CAF CLI, CML, graph, Studio, or receipt service is required to select, execute, verify, or close work. The project should nevertheless keep clean machine-readable boundaries—Beads JSON output, stable external references, manifests, receipts, repository locks, and task packs—so CAF can later ingest the project without replacing or duplicating Beads.

### 2.5 genes-ts assessment

The attached `genes-ts` snapshot is version 1.11.0 and requires Haxe 4.3.7 or later. Its useful capabilities include:

- split ECMAScript module (ESM) TypeScript output;
- `.ts` and `.tsx` emission;
- strict TypeScript-oriented output;
- typed import helpers;
- target-polymorphic `undefined` and `unknown` abstractions;
- React TSX authoring support;
- async/await support;
- source-map support;
- classic Genes JavaScript output as a second mode;
- snapshot, TypeScript, Node, browser, and Playwright-oriented testing guidance.

The experimental `ts2hx` path is useful but not a full Gutenberg converter. Its own plan states that it is JS-centric, TSX is not yet a core focus, some constructs are accepted only in high-signal forms, and narrow `Dynamic` or raw JavaScript escape hatches are expected. The emitter still contains explicit unsupported fallbacks.

Decision:

- use `ts2hx` for inventory, initial scaffolding, and selected leaf packages;
- never treat its output as accepted merely because it compiles;
- require source-level review and differential runtime tests;
- improve generic missing features in genes/ts2hx rather than adding WordPress-only hacks;
- keep a limitation ledger with minimized fixtures.

### 2.6 Haxe 4.3.7 PHP target assessment

The stock target provides important low-level tools:

- `php.Syntax.code` and other PHP-language expressions;
- native PHP array abstractions;
- PHP globals and superglobals;
- `php.Ref<T>` and generator-level reference handling;
- PHP-specific metadata;
- file-per-type class generation;
- configurable output front file, library directory, and namespace prefix;
- post-generation macro callbacks;
- access to the compiler source in `genphp7.ml` when a fork is justified.

Important constraints:

- `php.Ref<T>` is documented for externs, although the generator recognizes reference semantics more broadly. Authored-method behavior must be tested before relying on it.
- `@:phpGlobal` affects calls to extern static fields; it does not emit thousands of global function definitions.
- static initialization is emitted as class initialization, not arbitrary original-file top-level code.
- the normal output shape is Haxe runtime classes plus a bootstrap, not a WordPress distribution tree.
- traits and exact original-script behavior need explicit treatment.

### 2.7 Lessons from the two prior port projects

The OpenCodeHX and codex-hxrust snapshots contain practices that should be reused directly:

- immutable upstream and compiler pins;
- generated source inventories;
- explicit runtime seam classification;
- upstream behavioral oracles;
- compiler limitation ledgers;
- gate-driven milestones;
- Beads epics and dependency-aware ready queues;
- no-framework guardrails until parity-critical foundations are stable;
- parity scorecards and explicit pivot criteria;
- typed interop boundaries;
- fixtures and schema fingerprints;
- modes such as helper, sidecar, adapter, and replacement rather than premature replacement claims.

Therefore, the PHP reference, class, and generated-shell strategy remains a source-supported design hypothesis until Gate F1–F7 compile and run it.

## 3. Product vision

The long-term product is a WordPress-compatible publishing platform whose executable knowledge is concentrated in Haxe:

- Haxe’s type system captures contracts that are implicit in PHP and loosely represented in JavaScript.
- Macros and compiler passes capture repetitive compatibility knowledge deterministically.
- Agents work against stable, high-density Haxe APIs instead of repeatedly rediscovering low-level implementation details.
- Generated PHP and JavaScript retain compatibility with the WordPress ecosystem.
- Once behavior is captured, selected subsystems can be improved, generalized, or compiled to new targets without redoing the entire semantic discovery process.
- Future orchestration layers, including CAF, can later consume Beads, manifests, task packs, and receipts without changing the port’s authority model.

The near-term product is deliberately narrower:

> Produce a reproducible, testable experimental WordPress 7.0 distribution in which progressively more PHP and browser runtime logic is authored in Haxe, while preserving WordPress behavior and interfaces.

A generated WordPress distribution with:

- unchanged static assets where no executable logic is involved;

## 6. Goals

### 6.1 Product goals

1. Generate an installable WordPress 7.0.0-compatible distribution from Haxe-authored runtime logic.
2. Preserve WordPress public PHP APIs, file topology, hooks, data formats, HTTP behavior, and user experience.
3. Preserve Gutenberg package APIs, browser behavior, editor UX, package exports, and WordPress asset registration behavior.
4. Port all executable PHP included in the vanilla distribution, including bundled libraries and default-theme PHP.
5. Port all shipped JavaScript/TypeScript logic and, in the final scope tier, the source logic of bundled browser vendors.
6. Port all Gutenberg workspace packages, including runtime, build, and test packages, through staged completion tiers.
7. Make unsupported compiler behavior explicit and improve Haxe/genes generically when possible.
8. Make every source file, package, artifact, API, and test traceable through manifests.
9. Make the program safe for many agents through dependency-aware tasks, path ownership, small worksets, and verified close.
10. Prove a recoverable Codex + Beads workflow for a very large, multi-agent port while keeping future CAF ingestion possible.

### 6.2 Engineering goals

- Haxe 4.3.7 is pinned and reproducible.
- The PHP output layer can preserve native WordPress semantics without hand-edited PHP.
- genes-ts output typechecks under the project’s strict profile.
- Generated output is deterministic.
- Upstream-to-port source mapping is machine-readable.
- Compiler changes have minimized generic tests.
- A full build can be reproduced from a clean checkout.
- A failed parity check points to a bounded package, API, hook, request, or artifact.

### 6.3 Knowledge-density goals

- Repeated WordPress patterns become typed Haxe APIs or macros only after their behavior is understood.
- Public APIs include concise semantic documentation and compatibility notes.
- Agents can retrieve a bounded task pack rather than loading the full WordPress/Gutenberg repositories.
- Graph and manifest projections permit package-, API-, and feature-level navigation.

## 7. Non-goals before parity

The following are explicitly deferred:

- A generalized multi-target CMS abstraction.
- A redesigned WordPress public API.
- Breaking plugin or theme compatibility to obtain cleaner Haxe.
- Replacing WordPress’s database schema or serialized storage formats.
- Replacing React, the DOM, PHP, MySQL, or the web server during the 7.0 parity line.
- Making generated PHP look hand-written when the difference is not observable or operationally useful.
- Porting every build tool to Haxe before the runtime distribution works.
- Introducing broad `tink_*` abstractions at public compatibility boundaries.
- Optimizing all output before the behavior is correct.
- Claiming production readiness, security equivalence, or drop-in replacement before all required gates pass.
- Adding another task database, Markdown task queue, or mandatory orchestration framework before evidence justifies it.

## 8. Scope model and definition of “100% Haxe”

### 8.1 Two baselines

The project must not conflate these baselines:

#### Baseline A: WordPress 7.0 distribution parity

Authority:

- exact WordPress 7.0.0 release commit/tag;
- exact Gutenberg commit pinned by the WordPress 7.0 package metadata;
- exact generated assets included in the official distribution;
- exact upstream test suites at those refs.

This baseline determines whether `wphx-distro` is a WordPress 7.0-compatible distribution.

#### Baseline B: forward Gutenberg package parity

Authority:

- attached Gutenberg 23.4.0 snapshot, replaced at Gate G0 by its exact Git commit;
- all 123 package workspaces and other selected workspaces;
- package tests and build outputs at that ref.

This baseline tests broad genes-ts/ts2hx capability and creates a path toward later WordPress releases. It may move independently after the 7.0 release line is frozen.

### 8.2 Completion tiers

| Tier | Meaning | Required claim |
|---|---|---|
| C0 | Harness, inventory, and mixed upstream/Haxe walking skeleton | “Experiment bootstrapped” |
| C1 | WordPress first-party PHP runtime logic for the official distro is Haxe-authored | “Core PHP parity candidate” |
| C2 | WordPress/Gutenberg first-party shipped browser runtime logic is Haxe-authored | “First-party distro parity candidate” |
| C3 | Bundled PHP and browser vendor runtime logic is Haxe-authored or explicitly replaced by a behavior-compatible Haxe implementation | “Vanilla runtime logic is Haxe-authored” |
| C4 | Gutenberg build, test, dev, routes, widgets, and other workspace logic is Haxe-authored | “Gutenberg monorepo logic ported” |
| C5 | No hand-authored PHP/JS/TS/TSX implementation remains outside upstream oracles and approved temporary tool bootstrap | “100% Haxe-authored executable knowledge” |

A tier is not achieved by line-count percentage. Every applicable manifest entry and critical gate must pass.

### 8.3 What may remain non-Haxe

The following may remain in their native formats because they are data or target artifacts rather than authored runtime logic:

- CSS, SCSS, images, fonts, SVG, JSON schemas, localization catalogs, and static HTML;
- generated PHP, TypeScript, TSX, JavaScript, source maps, and asset manifests;
- package metadata and build configuration;
- SQL fixture/data files;
- upstream oracle repositories;
- temporary third-party bootstrap tools explicitly recorded as migration debt.

SCSS build logic written in JavaScript is executable and is in C4/C5 scope even though SCSS source files themselves are not translated.

### 8.4 Vendor policy

“Everything in the vanilla distribution” includes bundled vendor code. It does not mean reimplementing PHP extensions, browsers, React, MySQL, or the operating system.

For each bundled vendor, choose one recorded strategy:

1. direct Haxe port preserving the vendor API;
2. Haxe reimplementation backed by a host primitive, with a fallback preserving supported environments;
3. generated wrapper around an external runtime dependency, only when the official WordPress distribution has the same dependency assumption;
4. temporary upstream artifact exception with a removal gate.

No vendor disappears from the inventory merely because it is large or minified.

## 10. Compatibility contract

Compatibility is a set of observable contracts, not merely equal return values in isolated functions.

### 10.1 PHP API contract

The port must preserve, where present upstream:

- global function names;
- declaration timing and conditional availability;
- parameter order;
- required/optional status;
- default values and their exact PHP forms when reflection exposes them;
- by-reference parameters and return values;
- variadics;
- accepted callable shapes;
- return value types and false/null/error conventions;
- global class, interface, trait, and constant names;
- inheritance and implemented interfaces;
- public/protected/private method and property visibility;
- static versus instance behavior;
- magic methods;
- dynamic property behavior where WordPress supports it;
- reflection-visible names and native class identity;
- serialization and JSON behavior;
- PHP warning, notice, deprecation, exception, and fatal-error classes when externally observable.

### 10.2 File and load contract

The generated distribution must preserve:

- expected original paths;
- bootstrap entry files;
- `ABSPATH`, `WPINC`, and related path behavior;
- `require`/`include` order;
- `require_once` and `include_once` identity;
- file-level side effects;
- conditional declarations;
- `pluggable.php` behavior;
- `SHORTINIT` behavior;
- multisite bootstrap variants;
- direct-load guards;
- template output order;
- include return values when used;
- caller-scope inputs and required caller-visible mutations;
- source path behavior needed by `__FILE__`, `__DIR__`, stack traces, or diagnostics.

The generated shell at an original path is part of the compatibility implementation, not a workaround to hide.

### 10.3 Native PHP value contract

At the WordPress boundary, preserve native PHP semantics for:

- mixed-key arrays;
- key coercion;
- insertion order;
- references and aliasing;
- copy-on-write effects where observable;
- `isset`, `empty`, null, false, and missing-key distinctions;
- numeric/string coercion;
- object-to-array behavior;
- native resources;
- callbacks represented as strings, arrays, closures, and invokable objects;
- serialized strings and object class names.

Do not expose Haxe `Array`, anonymous-object wrappers, or map wrappers at public PHP boundaries unless the upstream contract itself uses an equivalent object.

### 10.4 Hook contract

For every action and filter:

- preserve hook name;
- preserve registration and execution order;
- preserve priority ordering and stable tie behavior;
- preserve callback identity/removal behavior;
- preserve accepted-argument count;
- preserve reference mutation where applicable;
- preserve current-filter/action stack behavior;
- preserve all/doing_action/did_action behavior;
- preserve exceptions/fatals and short-circuit behavior.

Hook traces are first-class oracle artifacts.

### 10.5 Data contract

Preserve:

- database schema and upgrade numbers;
- SQL behavior relevant to supported databases;
- option, transient, metadata, post, term, user, and comment formats;
- serialized data;
- object cache keys and invalidation behavior;
- REST schemas and payloads;
- XML-RPC, feeds, and export/import formats;
- cron schedules and stored events;
- passwords, nonces, cookies, sessions, and capabilities;
- file and media metadata;
- block grammar and rendered markup.

A site must be able to alternate between the vanilla and Haxe distributions against a copied database during compatibility testing without a one-way data migration.

### 10.6 HTTP and process contract

Preserve:

- status codes;
- headers, order where relevant, and duplicate-header behavior;
- cookies;
- redirects;
- body bytes after documented normalization;
- output buffering;
- compression interaction;
- REST and admin AJAX behavior;
- CLI/process exit codes for included command surfaces;
- mail and cron side effects through test adapters.

### 10.7 Browser and package contract

Preserve:

- WordPress script handles and declared dependencies;
- global symbols such as the expected `wp.*` package surfaces;
- module exports;
- package names and export maps;
- package side effects;
- React component and hook behavior;
- DOM event behavior;
- editor data stores, selectors, actions, controls, and subscriptions;
- i18n extraction and runtime translation;
- block registration and serialization;
- CSS class names and markup relied on by themes/plugins;
- keyboard, focus, accessibility, and iframe behavior;
- browser console warning/error behavior relevant to consumers;
- source maps and debuggability.

### 10.8 User-experience contract

The goal is not merely that pages load. Critical flows must be behaviorally and visually equivalent:

- install and upgrade;
- login/logout/password reset;
- create, edit, preview, publish, schedule, and revise content;
- media upload/edit/select;
- taxonomy, user, comment, menu, widget, and theme operations;
- site editor and post editor;
- plugin and theme activation/deactivation/update;
- multisite network administration;
- localization and right-to-left rendering;
- accessibility navigation and screen-reader semantics.

### 10.9 Experimental safety contract

Until the replacement gate passes:

- identify the build as experimental in admin and generated metadata;
- do not allow vanilla core auto-update to overwrite the generated distribution silently;
- preserve plugin/theme update behavior unless a test profile disables network access;
- provide an explicit rollback path to the same upstream WordPress version;
- keep database backups in integration/e2e workflows;
- never run destructive compatibility tests against production data.

### 11.4 Internal versus public representation

Use two representation zones:

#### Public compatibility zone

- native PHP arrays and native JS objects;
- exact public names;
- exact nullable/undefined/false distinctions;
- target callables and resources;
- minimal wrappers;
- no speculative abstraction.

#### Typed implementation zone

- Haxe classes, enums, abstracts, typedefs, and functions;
- narrow domain types;
- pure functions where behavior allows;
- optional `tink_*` use only when generated behavior is equivalent and measured;
- adapters at the boundary.

A conversion between zones must be explicit and testable. Avoid repeatedly converting large arrays/objects in hot paths.

### 11.5 Target-specific source layout

Use target-local code openly:

```text
src/php/...     # may use php.Syntax, NativeArray, Ref, native PHP externs

src/shared/...  # only code already demonstrated to have equivalent semantics
```

This is preferable to a false shared layer full of `Dynamic` and runtime branching.

## 12. Haxe PHP target design

### 12.1 Use the stock target as the implementation emitter

The stock Haxe PHP target should initially emit internal implementation types under a private namespace such as:

```text
WPHX\Impl\...
WPHX\Runtime\...
```

Avoid placing all Haxe-generated implementation types in WordPress’s global namespace. Only public WordPress classes/interfaces that must have exact native identities should be emitted there.

Recommended compiler settings include pinned values for:

- output directory;
- PHP library directory;
- namespace prefix;
- source maps in debug/parity profiles;
- dead-code elimination profile, validated carefully because reflection and callbacks can make reachability non-obvious.

### 12.2 Macro responsibilities

Macros should own typed, compiler-aware work:

| Responsibility | Mechanism |
|---|---|
| Register a WordPress source unit | class/module metadata or build macro |
| Describe global functions | `@:wp.global(...)` metadata and generated manifest |
| Describe exact public classes | native-name metadata plus ABI manifest |
| Generate repetitive Haxe façades | `Context.defineModule` / build macros |
| Validate signatures | typed AST inspection |
| Track native-array/ref use | type inspection and lints |
| Attach source provenance | generated metadata/constants |
| Emit build manifest | macro callback before/after generation |
| Trigger linker | build command or post-generation callback |
| Reject unsupported public shapes | compile-time errors |

Macros should not perform broad textual rewrites of generated PHP.

### 12.3 WordPress ABI façade

A Haxe implementation might be authored as:

```haxe
package wphx.wordpress.hooks;

import php.NativeArray;

@:wp.source("src/wp-includes/plugin.php")
class PluginApi {
  @:wp.global("add_filter")
  public static function addFilter(
    hookName:String,
    callback:Dynamic,
    priority:Int = 10,
    acceptedArgs:Int = 1
  ):Bool {
    // Typed implementation using native PHP boundary values.
    return HookRegistry.add(hookName, callback, priority, acceptedArgs);
  }
}
```

The linker emits the WordPress-facing shell:

```php
<?php
// Generated by wphx. Do not edit.

if ( ! function_exists( 'add_filter' ) ) {
    function add_filter( $hook_name, $callback, $priority = 10, $accepted_args = 1 ) {
        return \WPHX\Impl\WordPress\Hooks\PluginApi::addFilter(
            $hook_name,
            $callback,
            $priority,
            $accepted_args
        );
    }
}
```

This preserves the global name and declaration timing while keeping implementation logic in Haxe.

### 12.4 Signature source of truth

Do not manually duplicate signatures in three places. The upstream PHP ABI extractor creates a canonical signature record; Haxe metadata binds an implementation to that record; the compiler validates compatible Haxe types; and the linker emits the PHP signature.

The ABI record must capture:

- namespace/global scope;
- function or method name;
- visibility/static/final/abstract flags;
- by-reference return;
- each parameter name, position, default source expression, by-reference flag, variadic flag, and type declaration;
- attributes/doc compatibility notes where relevant;
- enclosing conditional/load segment;
- source location and hash.

Default values should be emitted from a verified normalized PHP AST, not reconstructed from Haxe defaults when reflection or exact syntax matters.

### 12.5 References

References are a hard gate.

Required tests:

- authored Haxe methods with `php.Ref<T>` parameters;
- authored Haxe methods returning `php.Ref<T>`;
- wrapper-to-method propagation;
- aliasing of array elements and object properties;
- callback parameters passed by reference;
- global references;
- reference assignments and later mutation;
- variadic references where PHP permits them.

If `php.Ref` is reliable for authored methods, use it at boundary implementations. If not, add a minimal backend feature or emit the public implementation directly as a top-level PHP function from a specialized Haxe script emitter. A boxed `RefCell` is not an acceptable general public-boundary substitute because it changes PHP aliasing.

### 12.6 Global variables and superglobals

Public WordPress globals must remain real PHP globals. Do not centralize them in a Haxe object visible only through wrappers.

Provide target-specific typed accessors that compile to direct operations:

```haxe
var wpFilter:NativeArray = WpGlobals.get("wp_filter");
WpGlobals.set("wp_filter", wpFilter);
```

Conceptual output:

```php
$GLOBALS['wp_filter']
```

Required support:

- reads and writes;
- existence checks;
- references to globals;
- global declarations inside generated script segments;
- direct access to `$_GET`, `$_POST`, `$_REQUEST`, `$_SERVER`, `$_COOKIE`, `$_FILES`, and `$_ENV`;
- test isolation/reset helpers that do not affect production behavior.

### 12.7 Native arrays

Use `php.NativeArray`, `NativeAssocArray`, `NativeIndexedArray`, and `NativeStructArray` intentionally. Create WordPress-specific abstracts only when they add static guarantees without changing runtime representation.

Examples:

- `WpArgs` may be an abstract over a native associative array;
- `HookCallbacks` may be an abstract over the exact nested native-array shape;
- a public `array` parameter must not silently become a Haxe `Array`;
- a `WP_Error` payload must preserve upstream array/object shape.

Add conformance tests for numeric-string keys, sparse arrays, insertion order, union/merge behavior, `isset` versus key existence, references, sort functions, and serialization.

### 12.8 Public classes

Evaluate direct Haxe emission with exact native names:

```haxe
@:native("WP_Query")
class WPQuery {
  // ...
}
```

Gate this through fixtures for:

- exact `get_class` and `is_a` values;
- reflection names, methods, properties, defaults, and visibility;
- inheritance and interfaces;
- magic methods;
- dynamic properties;
- clone behavior;
- serialization/unserialization;
- callable methods;
- exception traces;
- native extension interfaces such as iterators or array access.

If the stock target introduces incompatible public-class behavior, retain Haxe-authored bodies but emit the public class through the specialized backend/linker.

### 12.9 Interfaces and traits

- Haxe interfaces should map directly where the generated PHP contract matches.
- PHP traits require a dedicated strategy because Haxe has no general trait construct.
- The initial inventory contains only seven traits, so implement `@:wp.phpTrait` metadata and linker/backend emission rather than pretending composition is always equivalent.
- Classes using traits must preserve conflict resolution and aliases.

### 12.10 Conditional declarations and pluggable APIs

A class autoloader cannot replace conditional global declaration behavior.

The file-segment manifest must preserve ordered constructs such as:

1. guard/constant check;
2. class include;
3. conditional global function declaration;
4. top-level initialization;
5. inline template output;
6. later declaration;
7. file return.

`pluggable.php` must declare a function only when a plugin has not already supplied it. The generated shell owns that declaration condition; the Haxe implementation class may already be loadable without making the global function exist.

### 12.11 Mixed PHP/HTML and caller scope

The 653 mixed files are a separate migration class, not ordinary modules.

Represent each original script as ordered segments:

- declaration segment;
- executable Haxe script segment;
- literal output segment;
- template expression segment;
- include/require segment;
- return/exit segment.

Three implementation modes are allowed:

1. **Renderer mode:** a Haxe function receives a native context and returns/prints output. Use only when caller-local mutation is not observable.
2. **Context bridge mode:** the shell captures required locals, invokes Haxe, then applies explicitly declared outputs. Use when the local contract is bounded.
3. **Direct script-emission mode:** a target-specific backend emits Haxe-authored statements into the original PHP scope. Use when arbitrary caller scope, references, or interleaving make a static-method call incompatible.

The inventory tool must classify template inputs/outputs and block a false “done” status when scope behavior is unknown.

### 12.12 Includes and top-level side effects

The generated shell, not a centralized boot class, should initiate original-file side effects at the original load point. Implementation can call Haxe static functions, but the shell preserves when they run.

Add explicit fixtures for:

- repeated include versus include_once;
- circular includes;
- relative path resolution;
- include inside a function/method scope;
- variables created by an included file;
- include return values;
- direct web access;
- `ABSPATH` guards;
- nested conditional function declarations.

### 12.13 Errors, output, and process control

Provide target-specific APIs for:

- `trigger_error` and WordPress deprecation helpers;
- native warnings/notices where upstream emits them;
- headers and cookies;
- output buffering;
- `echo`/print and binary output;
- `exit`/`die` with correct code/body;
- shutdown functions;
- error handlers and exception handlers;
- backtrace normalization for tests.

Haxe exceptions must not accidentally replace upstream false/`WP_Error`/warning behavior.

### 12.14 Linker responsibilities

`wphx-link` is a deterministic compiler stage, not an ad hoc copy script. It should:

- consume Haxe compiler output plus typed manifests;
- validate that all referenced implementation symbols exist;
- generate global wrappers and public files;
- generate exact file paths and load order;
- copy non-executable assets with checksums;
- integrate browser artifacts;
- emit build/provenance manifests;
- reject unresolved source units or unapproved exceptions;
- support debug and release profiles;
- produce byte-identical output from identical inputs.

### 12.15 When to fork `genphp7.ml`

Create a backend task only when all are true:

1. a minimized generic fixture demonstrates the blocker;
2. ordinary Haxe and `php.Syntax` cannot express it safely;
3. a generated shell cannot preserve semantics or has unacceptable overhead;
4. the feature benefits procedural PHP ports beyond a single WordPress call site;
5. the change has compiler-level tests;
6. the fork is pinned and its delta is documented.

Candidate backend features, subject to spikes:

- direct global function emission from metadata;
- top-level/script module emission;
- authored by-reference methods and returns as a supported contract;
- PHP trait emission/use;
- reduced runtime mode for procedural applications;
- exact public property/default emission;
- source-path controls for diagnostics;
- a structured post-emission hook rather than textual rewriting.

Do not fork merely to change indentation, file names that the linker can handle, or internal namespaces.

### 12.16 “Idiomatic PHP output” priority

Priorities are:

1. behavior;
2. public ABI and file contract;
3. security;
4. debuggability and source mapping;
5. performance and deployment footprint;
6. readable/idiomatic generated PHP;
7. cosmetic similarity to upstream.

Idiomatic output matters where it improves stack traces, operations, audits, or plugin reflection. It is not a reason to delay a correct generated wrapper.

## 13. genes-ts and Gutenberg design

### 13.1 Primary browser strategy

Use Haxe as canonical source and genes-ts TypeScript/TSX as the primary auditable intermediate artifact.

Each package must pass:

1. Haxe compilation;
2. deterministic genes-ts generation;
3. strict TypeScript typecheck;
4. package build/bundle;
5. upstream unit/fixture tests adapted to the generated package;
6. differential runtime tests;
7. package API and export comparison;
8. browser/e2e tests where applicable.

### 13.2 Classic Genes fallback

Classic Genes JavaScript is retained for three purposes:

- isolate whether a failure is in Haxe logic, TypeScript emission, or downstream TypeScript tooling;
- provide a temporary runtime path when genes-ts lacks a feature but classic Genes is correct;
- compare generated runtime behavior.

A package is not considered genes-ts-complete while it relies on the classic fallback. The exception remains visible in `compiler-pressure.jsonl` and the package parity record.

### 13.3 `ts2hx` policy

`ts2hx` may:

- create initial file/module scaffolds;
- translate straightforward utility code;
- generate extern candidates;
- identify unsupported syntax;
- bootstrap package API typedefs;
- provide a before/after migration trace.

It may not:

- decide semantic correctness;
- silently replace types with broad `Dynamic` at public APIs;
- inject package-specific raw JavaScript without a ledger entry;
- mark a source unit complete;
- bypass React/TSX review;
- determine portability.

Every generated scaffold starts in `scaffolded`, not `ported`.

### 13.4 Package public contract

For every `@wordpress/*` package, capture:

- package name/version/private status;
- entry points and export map;
- ESM/CommonJS expectations;
- TypeScript declarations;
- side-effect declarations;
- internal and external dependencies;
- WordPress global exposure;
- script handles and asset metadata;
- public values, functions, classes, React components, hooks, and types;
- i18n text domains;
- CSS/SCSS assets;
- test suites and browser surfaces.

Use API Extractor or an equivalent deterministic declaration fingerprint where appropriate. Runtime exports must also be enumerated because TypeScript declarations alone do not prove behavior.

### 13.5 React and TSX

Use the genes-ts React/TSX path, but prove these behaviors early:

- JSX runtime mode;
- fragments;
- component statics and display names;
- hooks and rules-of-hooks compliance;
- refs and callback refs;
- context;
- portals;
- error boundaries;
- suspense/lazy boundaries used upstream;
- React event typing and runtime behavior;
- prop spreading and `undefined` versus absent props;
- children normalization;
- memo/forwardRef behavior;
- source-map stack traces.

Do not introduce a Haxe component framework that changes output semantics during the parity line. A thin HXX/TSX authoring layer is acceptable when it emits the same React model.

### 13.6 JS object semantics

Use target-minded Haxe for JavaScript boundaries:

- distinguish `undefined`, `null`, false, and missing properties;
- preserve object spread and property order where observable;
- preserve prototypes and `instanceof` where APIs expose them;
- preserve symbols, iterables, promises, async ordering, and microtasks;
- use externs for DOM and npm APIs;
- avoid converting every object to a Haxe class;
- use `Dynamic` only at measured dynamic boundaries with tests.

### 13.7 WordPress package integration

The build profile must emit both modern module artifacts and WordPress-compatible assets:

- package modules;
- `wp.*` global exposures where upstream does so;
- dependency metadata (`*.asset.php` or the exact current equivalent);
- script handle mapping;
- translations and text-domain metadata;
- source maps;
- minified and development builds;
- module registrations introduced by the WordPress baseline;
- exact copied file locations expected by the Core Gutenberg integration scripts.

### 13.8 Package-wave order

Use the inspected dependency graph as a starting order:

- **Wave J0:** compiler profile, externs, package harness, export fingerprints.
- **Wave J1:** pure or mostly pure leaf utilities such as escaping, entities, hooks, shortcode, URL, word count, priority queue, and parser primitives.
- **Wave J2:** i18n, deprecated, element, DOM, date, API fetch, keycodes, route.
- **Wave J3:** compose, data, icons, router, preferences persistence, React i18n.
- **Wave J4:** abilities, data controls, rich text, UI, components, blocks.
- **Wave J5:** core data, format library, notices, plugins, preferences, interface.
- **Wave J6:** block editor.
- **Wave J7:** block library, including its PHP render side through the PHP pipeline.
- **Wave J8:** editor and global styles.
- **Wave J9:** edit-post, edit-site, edit-widgets, customize-widgets, fields, block directory.
- **Wave J10:** boot, build, edit-site-init, routes, widgets, and integration roots.
- **Wave J11:** build/test/dev packages and remaining workspaces.
- **Wave J12:** bundled browser vendors used by the vanilla distribution.

Individual packages may move earlier for a vertical slice, but dependencies and test fixtures must remain explicit.

### 13.9 Compiler-pressure discipline

A compiler-pressure record contains:

```json
{

  "id": "genes:tsx:callback-ref-001",
  "tool": "genes-ts",
  "upstreamRefs": ["gutenberg:<sha>:packages/components/..."],
  "minimalFixture": "fixtures/genes/callback-ref",
  "observed": "...",
```

## 14. Porting methodology

### 14.1 Inventory before translation

Generate authoritative inventories from full checkouts:

- all source files and generated/distribution artifacts;
- PHP functions, classes, interfaces, traits, constants, globals, includes, and script segments;
- PHP dynamic-feature and compatibility-risk tags;
- JS/TS modules, imports, exports, package edges, runtime environments, and syntax features;
- build pipelines that transform Gutenberg into WordPress Core assets;
- upstream tests and fixtures;
- shipped vendor provenance;
- licenses and notices.

The inventory digest is a release input. A new upstream ref invalidates affected task packs and parity states.

### 14.2 Classify every source unit

Suggested source-unit classes:

- pure algorithm;
- native-boundary adapter;
- public API façade;
- data model;
- procedural script;
- bootstrap/load script;
- template/mixed output;
- generated registry;
- vendor library;
- build tool;
- test/fixture;
- static data/asset.

Suggested risk tags:

- globals;
- references;
- native arrays;
- conditional declarations;
- includes;
- caller scope;
- reflection;
- serialization;
- dynamic calls;
- hooks;
- database;
- filesystem;
- networking;
- process/output;
- security-sensitive;
- browser/DOM;
- React/TSX;
- async;
- package cycle/side effect;
- performance-hot.

### 14.3 Porting unit size

A task should normally contain one of:

- one public API plus its private helpers;
- one PHP file cluster with a coherent load boundary;
- one small Gutenberg package;
- one bounded feature slice across a few packages;
- one compiler limitation;
- one test harness capability.

Avoid “port wp-admin,” “port Gutenberg,” or “port 100 files” tasks. Those are epics.

### 14.4 Agent port loop

For each unit:

1. Load its generated task pack.
2. Run upstream tests/oracle and store the baseline receipt.
3. Inspect public API, side effects, hooks, data shapes, and dependencies.
4. Generate or write a Haxe scaffold.
5. Implement the smallest behavior slice.
6. Compile both debug and parity profiles.
7. Inspect generated target output for accidental representation changes.
8. Run unit and differential tests.
9. Run affected upstream tests.
10. Add generic compiler fixtures for any compiler gap.
11. Update inventory, ABI/API, provenance, and parity manifests.
12. Emit a verification receipt.
13. Request review; close only through verified close.

### 14.5 Behavior-first translation style

During initial translation:

- preserve control flow where it carries subtle semantics;
- preserve public data shapes;
- preserve ordering;
- avoid clever refactors;
- add types around observed contracts;
- isolate dynamic behavior rather than eliminating it speculatively;
- document target-specific assumptions.

After a unit reaches integration parity, a separate refactor task may introduce more idiomatic or functional Haxe, with the same differential tests as a guard.

### 14.6 Vertical slices

Before broad waves, complete these slices:

#### Slice V1: PHP procedural kernel

- generated original-path file;
- one global API family;
- globals/native arrays/references;
- conditional declaration;
- upstream and plugin fixture tests.

Recommended family: hooks (`WP_Hook`, `add_filter`, `apply_filters`, removal, current hook stack).

#### Slice V2: WordPress short bootstrap

- `wp-load.php` and `SHORTINIT` path;
- constants/load order;
- database connection through `wpdb`;
- options/cache read and write;
- one fixture plugin.

#### Slice V3: content request

- create a post in the shared schema;
- `WP_Query` retrieval;
- theme/template rendering;
- headers and output comparison.

#### Slice V4: browser package

- one pure Gutenberg leaf package through ts2hx/Haxe/genes-ts;
- strict typecheck;
- package export parity;
- Node and browser differential tests.

Recommended candidates: `@wordpress/escape-html`, `@wordpress/html-entities`, or `@wordpress/wordcount` after confirming fixture quality.

#### Slice V5: React/editor component

- one component with hooks, refs, i18n, events, styling, and browser tests;
- generated TSX review;
- visual and accessibility comparison.

#### Slice V6: end-to-end authoring

- install the generated distro;
- log in;
- create and publish a post in the block editor;
- fetch it over REST;
- render it on the front end;
- activate a compatibility fixture plugin and theme.

### 14.7 Compiler versus application fix rule

Use this decision tree:

1. Is the behavior a WordPress-specific public convention? Put it in `wphx-*`.
2. Is it a PHP/TypeScript language feature any Haxe user might need? Fix the target/compiler.
3. Is it a host library API? Add/fix an extern or target helper.
4. Is it generated-layout policy? Fix the linker/build profile.
5. Is it a one-off source bug in the Haxe port? Fix the application.
6. Is a temporary raw target escape required? Add a ledger item and removal gate.

### 14.8 No silent debt

Temporary upstream PHP/JS, raw target snippets, broad `Dynamic`, disabled tests, and unported files all require explicit exceptions. An exception has:

- owner;
- reason;
- exact paths;
- behavioral coverage;
- security impact;
- removal task;
- expiry milestone;
- release claim it blocks.

## 16. Multi-agent operating model

### 16.1 Single project-management home

The root Beads database is the only active task store for the program. Do not initialize independent Beads databases inside WordPress, Gutenberg, genes-ts, Haxe compiler, generated distribution, or task worktrees.

All task commands should resolve to the program database. When an agent is working from a nested repository or a worktree where automatic discovery is ambiguous, use `BEADS_DIR=<program-root>/.beads` or the project wrapper `tools/bd-wphx`. The wrapper must not implement its own task database; it only pins the root and invokes `bd`.

### 16.2 Worksets

Recommended worksets (implemented as Beads labels plus documented ownership, not as a second tracker):

- `ws:php-platform`
- `ws:wp-kernel`
- `ws:wp-content`
- `ws:wp-admin`
- `ws:wp-rest-network`
- `ws:wp-media`
- `ws:php-vendors`
- `ws:genes-platform`
- `ws:gutenberg-foundations`
- `ws:gutenberg-editor`
- `ws:gutenberg-apps`
- `ws:browser-vendors`
- `ws:oracle-parity`
- `ws:program-operations`

Each workset has one integration owner at a time. Package/file tasks may execute concurrently if their owned paths and generated outputs do not overlap.

### 16.3 Task packet contract

Every executable task description must include:

```text
Beads issue ID, stable `WPHX-*` external reference, and title
Parent epic, milestone, and gate
Why this unit exists
Exact upstream baseline/ref and paths
Exact Haxe-owned paths
Generated artifact paths
Public API/ABI entries
Dependencies and dependents
Risk tags
Out-of-scope boundaries
Oracle commands and expected baseline receipt
Required tests
Required manifest updates
Compiler-pressure escalation rule
Performance/security concerns
Acceptance criteria
Required receipt schema
Reviewer/integration owner
```

A task is not decision-complete when it says only “port package X.”

### 16.4 Claims, branches, and worktrees

- Pull Beads state before selecting work when a remote is configured.
- Select only from `bd ready --json`, unless an integration owner explicitly assigns blocked investigative work.
- Inspect with `bd show <id> --json` and claim atomically with `bd update <id> --claim` before editing.
- One active agent claim per task.
- One Git branch/worktree per task or tightly coupled task group.
- Branch name contains the Beads issue ID and preferably the `WPHX-*` external reference.
- The issue records owned paths, generated-output prefixes, acceptance commands, and expected receipt paths.
- Shared compiler/runtime files require a dedicated compiler-pressure issue or an integration-owner lock.
- Agents do not make opportunistic cross-domain refactors.
- Cross-repository changes use an explicit task packet and receipt-backed handoff.
- When work is abandoned, release or reassign the issue explicitly; do not leave stale `in_progress` claims.

### 16.5 Safe parallelism

Safe parallel work after platform contracts stabilize:

- independent Gutenberg leaf packages;
- independent bundled vendors;
- independent default themes;
- PHP domains with frozen boundary APIs;
- tests/fixtures for already specified APIs;
- inventory/provenance cleanup.

Keep these serial or tightly coordinated initially:

- global ABI metadata;
- linker/file-segment model;
- PHP native value/reference layer;
- WordPress bootstrap/load order;
- hook kernel;
- genes package/export conventions;
- React authoring conventions;
- parity schema;
- compiler forks.

### 16.6 Review requirements

A port review checks:

- upstream behavior and API, not only Haxe style;
- public native values;
- load/side-effect timing;
- generated target code at boundary points;
- exception/warning/error behavior;
- tests and normalization;
- manifest/provenance updates;
- compiler-pressure debt;
- performance in known hot paths;
- no accidental broad abstraction.

Compiler/backend changes require a second reviewer and generic fixtures.

### 16.7 Verified close

A task closes only when:

- acceptance tests pass from a clean worktree;
- affected upstream tests pass or approved deltas are recorded;
- differential evidence is attached;
- generated output is reproducible;
- manifests are current;
- no untracked exception exists;
- the verification receipt is machine-readable and references the exact issue, source digest, upstream digest, and toolchain digest;
- integration has landed or the issue explicitly represents an isolated artifact;
- the Beads issue contains or links the receipt path and final result.

Only then run `bd close <id> --reason "<verified result>"`. A Codex narrative that says “done,” a green local command without a receipt, or a committed diff without integration is not sufficient.

### 16.8 Context economy

Generate task packs under `build/task-packs/<beads-id>/` containing only:

- issue details, dependencies, external reference, and acceptance criteria;
- upstream source slices;
- API/ABI records;
- dependency summaries;
- relevant tests and traces;
- Haxe source slices;
- compiler-pressure records;
- exact commands;
- recent receipts from direct dependencies.

Codex receives repository-wide policy through root `AGENTS.md`, local rules through nested `AGENTS.md`, and task-specific context through the generated pack. It should widen context only when the issue dependency graph or observed behavior justifies it. This is the initial token-economy mechanism; no graph framework is required.

## 17. Testing and parity plan

### 17.1 Test pyramid

#### Layer T0: compiler conformance

Small target-language fixtures for:

- references;
- native arrays/objects;
- global names;
- class/reflection behavior;
- include/script behavior;
- errors/output;
- TypeScript emission;
- TSX/React;
- async and module behavior;
- source maps.

These run on every compiler or platform-layer change.

#### Layer T1: Haxe unit/property tests

Haxe-authored tests for pure and typed implementation behavior. Use property tests for parsers, formatting, serialization boundaries, and collection semantics where useful.

#### Layer T2: upstream fixture/unit tests

Run the upstream PHP and JS tests against the Haxe implementation through adapters that preserve the test’s intent. Do not mechanically rewrite all tests before the harness can call generated APIs.

#### Layer T3: differential tests

Execute the same input against vanilla and Haxe builds and compare structured observations.

#### Layer T4: ecosystem compatibility

Run fixture plugins/themes and a curated external corpus.

#### Layer T5: browser/end-to-end

Playwright or upstream e2e flows for install, admin, editor, media, frontend, and multisite.

#### Layer T6: security, fuzzing, performance, reliability

Continuous and release-gate suites.

### 17.2 Differential observation schema

For each request or command, capture as applicable:

- status/exit code;
- normalized headers and cookies;
- response body hash and structured DOM/JSON/XML form;
- PHP warnings/notices/deprecations/exceptions;
- hook sequence, priority, callback identity class, and arguments;
- database queries, normalized parameters, result counts, and transaction effects;
- object-cache reads/writes/deletes;
- option/meta/content mutations;
- filesystem writes;
- mail/HTTP/cron side effects through adapters;
- browser console output;
- network requests;
- data-store actions/selectors;
- accessibility tree and screenshots for selected flows;

Normalization rules are versioned and reviewed. They may remove random IDs, timestamps, paths, and nondeterministic ordering only when the upstream contract does not require them.

- admin complexity;
- database/cron/REST usage;

Record why each item is present. A corpus pass is not proof of full compatibility, but failures reveal real contracts missing from synthetic fixtures.

### 17.9 Performance budgets

Initial proposed investigation budgets relative to the same vanilla build and environment:

- front-end cached and uncached request p50: no more than 10% slower;
- request p95: no more than 15% slower;
- bootstrap peak memory: no more than 15% higher;
- admin/editor server request p95: no more than 15% slower;
- critical browser interaction latency: no more than 10% regression;
- per-handle compressed JS size: no more than 5% larger without an accepted reason;
- no unbounded wrapper conversion or autoload growth.

These are proposed engineering budgets, not claims about current performance. Gate G3 measures the baseline and adjusts budgets only through an ADR.

### 17.10 Security gates

Security-sensitive domains include:

- escaping and KSES;
- SQL preparation;
- capabilities and roles;
- nonces;
- cookies/passwords/application passwords;
- file uploads and archive extraction;
- path traversal and symlinks;
- HTTP redirects and SSRF protections;
- REST/AJAX authorization;
- serialization/object injection;
- updates/signatures;
- editor sanitization.

Requirements:

- fuzz/property tests for parsers and sanitizers;

- software bill of materials (SBOM) for release artifacts;

## 18. Codex + Beads workflow

### 18.1 Operating model

The initial operating loop has four authorities:

| Surface | Authority |
|---|---|
| Codex + `AGENTS.md` | execution behavior and repository-local instructions |
| Beads | tasks, dependencies, priorities, claims, discoveries, and durable agent memory |
| Git/worktrees | source isolation, review, history, and integration |
| `wphx` manifests/receipts | coverage and verified actuality |

No other tool is required to start, continue, or recover the project.

### 18.2 Beads storage and safety model

Current Beads uses a Dolt-backed, version-controlled database. Therefore:

- use `bd` for all reads and writes;
- pin the exact Beads version in `toolchain.lock.json` because the CLI evolves;
- treat `.beads/issues.jsonl` and `bd export` as review/interchange outputs, not the canonical database or complete backup;
- use `bd dolt push` / `bd dolt pull` for configured remote synchronization;
- use `bd backup init`, `bd backup sync`, and restore drills for recoverable backups;
- never infer that committing an exported JSONL file alone preserves branches, history, working state, memory, or non-issue tables;
- keep one root database for this multi-repository program.

### 18.3 Bootstrap commands

From the new program root:

```bash
# Human bootstrap may use interactive `bd init`; agents use quiet mode.
bd init --quiet

# Install or refresh the Codex-specific integration, guidance, and hooks.
bd setup codex

# Review every generated change before committing it.
git status --short
git diff -- AGENTS.md .beads .codex

# Load workflow context and inspect unblocked work.
bd prime
bd ready --json
```

Record `bd --version` in `toolchain.lock.json`. Configure and test a Dolt remote when collaboration begins. Configure a backup destination before importing a large task graph.

`bd init` or `bd setup codex` may update `AGENTS.md`; the generated guidance must be merged with this project’s rules rather than accepted blindly.

### 18.4 Codex instruction layout

Codex reads `AGENTS.md` from the project root down to the current directory, with nearer instructions taking precedence. Use that behavior deliberately:

```text
AGENTS.md                         global invariants and command entry points
src/php/AGENTS.md                 PHP target, ABI, native-array/reference rules
src/js/AGENTS.md                  Genes, TSX, React, and JS-boundary rules
packages/wphx-link/AGENTS.md      deterministic linker rules
packages/wphx-oracle/AGENTS.md    oracle/normalization rules
test/AGENTS.md                    verification and no-test-weakening rules
```

Keep the root file concise. It should point to this PRD and runbooks instead of embedding them. Use `AGENTS.override.md` only for an intentional temporary/local override, and audit it before committing. Restart the Codex session after changing instructions because instruction discovery happens at session start.

Minimum root rules:

```markdown
This project uses Beads (`bd`) as the only task tracker.

- Run `bd prime` and `bd ready --json` at session start.
- Inspect with `bd show <id> --json`; claim with `bd update <id> --claim`.
- Track discovered work with a new issue linked by `discovered-from`.
- Use `bd remember` for durable non-task insights; do not create MEMORY.md.
- Do not use Markdown TODO lists as task state.
- Do not edit generated PHP/TS/JS or Beads storage by hand.
- Do not close an issue before its verification receipt and required manifests exist.
- Read `docs/prd/wordpress-haxe-port.md` and the generated task pack for the claimed issue.
```

### 18.5 Stable PRD keys and Beads IDs

The tables in this document use stable logical references such as `WPHX-102`. Beads should retain its generated collision-resistant IDs.

For every seeded issue:

- set `external_ref` to the `WPHX-*` key;
- prefix the title with the same key for human scanning;
- store the parent epic with a Beads parent relationship;
- add explicit blocking dependencies separately;
- never use title ordering as execution ordering.

Example intent:

```bash
bd create "WPHX-100 — PHP target feasibility" \
  --type epic --priority 1 --external-ref WPHX-100 --json

bd create "WPHX-101 — Stock Haxe PHP smoke project" \
  --type task --priority 1 --external-ref WPHX-101 \
  --parent <php-epic-id> --json
```

A deterministic seeding tool should query by external reference, create missing issues, update safe specification fields, and refuse destructive replacement of active human/agent state.

### 18.6 Dependency semantics

Use requirement language:

```bash
# <issue> needs <depends-on>.
bd dep add <issue> <depends-on>
```

Examples:

```bash
# The include/load spike needs global façade and native-value spikes.
bd dep add <F5-issue> <F1-issue>
bd dep add <F5-issue> <F3-issue>

bd ready --json
bd blocked --json
bd dep tree <issue>
```

Parent-child hierarchy is not a substitute for blocking dependencies. Sibling children are parallel unless explicit dependencies sequence them.

Use a small, stable classification vocabulary:

- priorities: P0 security/data-loss/release blocker; P1 current gate blocker; P2 planned implementation; P3 non-blocking debt; P4 deferred exploration;
- types: `epic`, `task`, `bug`, `feature`, or `chore`;
- labels: baseline, area/workset, gate, target, and risk tags;
- `discovered-from` and `related` links do not block unless a separate blocking dependency is added.

Do not adopt Beads formulas, molecules, or gates during the initial bootstrap merely because they exist. First prove the plain issue/dependency/claim/close loop. WPHX-808 may introduce a repeatable construct only when it removes demonstrated friction and remains recoverable.

### 18.7 Daily Codex task loop

```text
1. Synchronize Beads when applicable.
2. Run `bd prime`.
3. Run `bd ready --json`.
4. Inspect one candidate and its dependency tree.
5. Claim it atomically.
6. Create/switch to its branch and worktree.
7. Generate or refresh its bounded task pack.
8. Run the upstream oracle and save baseline evidence.
9. Implement the smallest accepted slice.
10. Run required checks and inspect generated target output.
11. Create compiler-pressure or discovered-work issues instead of expanding scope silently.
12. Update manifests and emit a verification receipt.
13. Review the diff and integration status.
14. Link the receipt to the Beads issue.
15. Close with a concrete verified reason.
16. Synchronize Beads and Git.
```

Recommended command skeleton:

```bash
bd dolt pull                    # when a remote is configured
bd prime
bd ready --json
bd show <id> --json
bd update <id> --claim

tools/wphx task-pack generate <id>
tools/wphx doctor
tools/wphx test task <id>
tools/wphx receipt verify --task <id>

bd close <id> --reason "Verified: <receipt-path>"
bd dolt push                    # when a remote is configured
```

### 18.8 Discovered work and durable memory

Create a new task when the finding requires action outside current acceptance:

```bash
bd create "WPHX-COMP — php.Ref authored return mismatch" \
  --type bug --priority 1 \
  --deps discovered-from:<current-id> \
  --description "Minimal fixture, observed/expected behavior, affected units, fallback, removal gate" \
  --json
```

Use `discovered-from` for provenance; add a blocking dependency only when the current or downstream task truly cannot proceed.

Use `bd remember` for durable facts that should appear in future `bd prime` context but are not actionable issues, for example a confirmed compiler quirk or repository convention. Do not use it as a substitute for an ADR, test, or task.

### 18.9 Multi-agent and worktree protocol

- An orchestrating agent or human selects independent ready tasks with non-overlapping owned paths.
- Codex subagents are optional. A write-capable subagent must map to one claimed Beads issue and an isolated worktree; analysis-only subagents may share read context but must not edit overlapping paths.
- The parent/orchestrating Codex session remains responsible for claim integrity, integration review, receipt verification, and close.
- Each worker claims its issue before receiving write authority.
- Each worker uses a dedicated Git worktree and branch.
- Shared generated manifests are updated through deterministic mergeable tooling; agents do not manually rewrite large shared files.
- Compiler/backend changes are isolated from application ports and receive a second reviewer.
- Every handoff records current commit, worktree/branch, commands run, remaining acceptance items, and receipt paths in the issue.
- Before integration, refresh Beads state and check that no task claim or dependency changed.
- If two agents discover the same work, merge/relate the Beads issues rather than maintaining duplicates.

For nested repositories or worktrees, use the root explicitly:

```bash
export WPHX_ROOT=/absolute/path/to/wordpresshx-port
export BEADS_DIR="$WPHX_ROOT/.beads"
# Or invoke the repository wrapper from any nested checkout:
tools/bd-wphx ready --json
```

The wrapper must resolve the program root robustly; a nested upstream Git root must not accidentally create or select another task database.

### 18.10 Receipt linkage and verified close

Implement `tools/wphx task close <id>` as a policy wrapper after the bootstrap phase. It should:

1. resolve the Beads issue and external reference;
2. reject blocked, stale, or unclaimed issues;
3. run or validate the task’s acceptance profile;
4. verify source/upstream/toolchain digests;
5. validate required manifest updates;
6. write a receipt under `receipts/`;
7. attach or record the receipt path on the issue;
8. call `bd close` only after every check succeeds.

The wrapper supplements Beads; it does not replace Beads state or invent a second workflow engine.

### 18.11 Sync, backup, and recovery

Before substantial task import:

```bash
bd backup init <durable-backup-path-or-supported-destination>
bd backup sync
```

For collaboration with a configured Dolt remote:

```bash
bd dolt pull
# perform task operations
bd dolt push
```

Gate G0 requires a documented restore drill into a clean checkout that reproduces:

- all issues and external references;
- dependencies and ready/blocked state;
- claims/status where applicable;
- durable memories;
- task audit/history required by the workflow;
- the same seeded-task idempotency result.

A human-readable `bd export` should be generated periodically for inspection and migration testing, but it is not accepted as the only backup.

### 18.12 Scaling rules

- Seed epics and foundation tasks first; generate leaf tasks only after inventories exist.
- Do not create one issue per source file until the platform contracts and task-packet generator are proven.
- Prefer one coherent API/file cluster/package per issue.
- Keep descriptions decision-complete but put large source context in generated task packs.
- Use labels or external refs for area/baseline/gate classification; do not encode all metadata into titles.
- Run `bd ready --json` and `bd blocked --json` in CI health checks after deterministic task imports.
- Audit orphaned children, duplicate external refs, stale claims, missing receipt links, and permanently blocked chains.
- Pin and upgrade Beads through dedicated issues with backup/restore tests.

### 18.13 Future CAF compatibility boundary

CAF is deferred. A later adoption must consume, not replace:

- Beads issue IDs, external references, dependencies, claims, and statuses through supported interfaces;
- repository and toolchain locks;
- source/package inventories;
- generated task packs;
- verification receipts and artifact provenance.

CAF must not require renumbering issues, moving task truth into CML, duplicating the Beads database, or changing `wphx` build/test commands. No `.cafetera` directory or CAF task is required before a separate adoption ADR is approved.

## 19. Portability and post-parity design

### 19.1 Current rule

Optimize for exact PHP/JavaScript behavior now. Portability is a later extraction, not an up-front tax.

### 19.2 Preserve future options cheaply

Do now:

- isolate target imports by source root;
- annotate host seams;
- keep pure algorithms pure when natural;
- use typed domain names instead of unexplained dynamic blobs;
- record native behavior relied upon;
- keep public boundary adapters thin;
- write differential tests independent of implementation;
- avoid unnecessary global state inside new internal APIs while preserving public globals.

Do not now:

- wrap every PHP function in a cross-target service;
- normalize PHP arrays into a universal collection;
- create a universal DOM/UI abstraction;
- replace callbacks/hooks with a different event model;
- add allocation-heavy conversions for theoretical targets;
- force `tink_*` types across plugin/package boundaries.

### 19.3 Refactor ratchet after parity

A subsystem may move toward shared/portable Haxe when:

1. its external compatibility tests are comprehensive;
2. its host dependencies are explicitly mapped;
3. a second target or consumer creates real demand;
4. the abstraction reduces total code/knowledge rather than adding adapters;
5. performance is measured;
6. generated PHP/JS behavior remains within contract.

### 19.4 Tink policy

Tink libraries can be valuable internally for typed errors, async, data structures, and functional composition. Introduce them case by case after the target behavior is proven.

Never replace a WordPress `false | value`, `WP_Error`, callback, native array, or Promise/undefined contract merely to make the Haxe API aesthetically uniform. A typed internal adapter can wrap it, but the public boundary remains exact.

## 21. Risk register and pivots

| Risk | Why it matters | Mitigation / pivot |
|---|---|---|
| Stock PHP target cannot emit exact procedural semantics | Core compatibility blocker | façade/linker first; minimal backend fork with fixtures |
| Authored `php.Ref` is unreliable | breaks APIs and hooks | F2; direct backend emission for reference boundaries |
| Public Haxe classes differ under reflection/serialization | plugin incompatibility | F4; specialized public-class emitter |
| Include/caller-scope semantics cannot be bridged cheaply | admin/theme templates break | file-segment model and direct script-emission mode |
| Haxe runtime overhead is high | request latency/memory | profile, reduce conversions, flatten hot wrappers, backend minimal-runtime work |
| DCE removes dynamically referenced symbols | callbacks/plugins fail | manifest-driven keep rules and dynamic-call fixtures |
| Haxe/native array conversion leaks | subtle data corruption | public native-value rule and property tests |
| genes-ts output does not support a Gutenberg construct | package blocks | generic compiler fixture/fix; temporary classic Genes fallback |
| `ts2hx` produces compiling but wrong code | false progress | scaffold-only status and differential tests |
| React/TSX output changes component behavior | editor UX regressions | early F9 plus visual/a11y/e2e tests |
| Attached Gutenberg baseline differs from WP 7.0 | wrong distro claim | strict dual-baseline lock and branch policy |
| Vendor scope dominates | program stalls | separate C3 wave; first-party gates remain explicit but no final “everything” claim |
| Agents collide in platform files | unstable merges | ownership locks, worksets, integration owners, compiler task queue |
| Task graph becomes thousands of vague items | coordination failure | generated decision-complete task packets and epic-level batching |
| Beads CLI/storage evolves during the program | scripts or recovery assumptions drift | pin version, use a thin wrapper, run upgrade tasks and restore drills |
| PRD, Beads, manifests, and receipts drift | stale/conflicting state | explicit authority matrix, stable external refs, generated validation |
| Upstream moves faster than port | permanent chase | freeze 7.0, separate forward line, automated delta inventory |
| Security patch latency | unsafe distribution | expedited workset; experimental-only until proven |
| Cleaner Haxe refactors change behavior | parity erosion | post-parity refactor tasks with same oracle tests |
| Output is hard to debug | operational friction | source maps, original-source metadata, stack normalizer, debug profile |
| Exact compatibility is impossible in a narrow edge case | blocks full claim | document exception, isolate generated direct target construct, keep removal task; never hide it |

### 21.1 Pivot rules

- **Macros-only → linker:** when original paths/global declarations are required.
- **Linker → PHP backend fork:** when top-level/reference/class semantics cannot be faithfully expressed or are too expensive.
- **genes-ts runtime → classic Genes temporary runtime:** when the Haxe logic is correct but TS emission is blocked.
- **automatic `ts2hx` → manual Haxe port:** when generated code becomes more expensive to verify than rewrite.
- **Direct `bd` commands → pinned `tools/bd-wphx` wrapper:** when CLI evolution or nested-repository discovery creates repeatable friction.
- **broad parallel port → serial kernel work:** when boundary contracts are still changing.

A pivot changes implementation strategy, not the parity objective.

## 25. Resolved defaults and questions for later evidence

### 25.1 Resolved by this PRD

- Exact compatibility outranks early portability.
- Haxe-authored logic may generate target-language compatibility files.
- Macros are necessary but not sufficient for WordPress PHP output.
- A deterministic ABI/file linker is part of the architecture.
- A minimal Haxe PHP backend fork is allowed only after feasibility evidence.
- genes-ts TypeScript/TSX is the primary browser experiment.
- `ts2hx` is an accelerator, not an authority.
- WordPress 7.0 and Gutenberg 23.4 are separate baselines.
- Beads is task truth; Codex + Beads is the complete initial workflow.
- Generated manifests and receipts are mandatory.
- Broad framework abstraction waits until parity.

### 25.2 Questions to answer through spikes, not speculation

- Does Haxe 4.3.7 reliably emit references for authored `php.Ref` methods and returns?
- Can direct `@:native` classes satisfy WordPress reflection/serialization/dynamic-property behavior?
- Which mixed templates require direct script emission rather than context bridging?
- How much Haxe Boot/runtime overhead remains after profiling and DCE?
- Which PHP backend additions are genuinely generic and upstreamable?
- Can genes-ts emit the exact package/module/TSX patterns used by critical Gutenberg packages without package-specific raw code?
- Which first packages give the best compiler coverage per unit of porting work?
- What percentage of the final vendor wave can use host primitives while preserving the official support matrix?

# ADR-001: PHP Emission Architecture

Status: Accepted

Date: 2026-06-20

## Context

WordPressHX is a complete WordPress port, not only a typed layer over upstream PHP. The port still has to preserve WordPress's public PHP contract while code is progressively moved into Haxe.

The stock Haxe 4.3.7 PHP target can emit and run Haxe implementation code, but WordPress also exposes behavior through:

- global functions with reflection-visible signatures and conditional declaration timing;
- by-reference parameters, by-reference returns, and callback reference mutation;
- native PHP arrays, globals, and superglobals;
- public global classes and interfaces;
- original file paths, ordered includes, top-level side effects, and include return values;
- mixed PHP/HTML templates with caller-scope locals;
- hook callbacks, native PHP callables, and native hook globals.

The PRD proposed escalation in this order:

1. ordinary Haxe;
2. target-specific Haxe APIs and macros;
3. deterministic ABI/file linker;
4. minimal Haxe PHP backend fork only for proven generic blockers.

WPHX-102 through WPHX-108 tested that proposal through F1-F7.

## Decision

Use this PHP emission architecture:

> Haxe-authored implementation code plus typed macro metadata and manifests, assembled into WordPress-facing PHP by a deterministic original-path ABI/file linker.

Do not fork the Haxe PHP backend for the current PHP feasibility evidence. WPHX-209 is not triggered by F1-F7.

The stock Haxe PHP target remains the implementation emitter for private Haxe-owned code. WordPress-facing PHP files are generated by the linker from typed manifests, upstream ABI records, and Haxe compiler output.

## Boundary Rules

### Ordinary Haxe

Use ordinary Haxe for implementation logic when PHP's public runtime shape is not itself the contract.

Examples:

- option normalization;
- scalar/string transforms;
- bounded class method bodies;
- deterministic helper payloads for generated shells;
- typed internal registries whose values are not public PHP aliases.

### Target-Specific Haxe APIs

Use target-specific Haxe APIs for typed access to PHP-native values where the representation must stay PHP-native.

Examples:

- `php.NativeArray` and native associative arrays;
- typed wrappers over `$GLOBALS` and superglobals;
- `php.Syntax.code(...)` only for narrow, reviewed target constructs;
- callback boundaries where PHP callables remain native.

Target-specific APIs must not hide public aliasing, references, or array shape changes.

### Macros

Macros own compiler-aware source and ABI metadata. They may:

- declare the upstream source unit a Haxe implementation belongs to;
- bind Haxe functions/classes to upstream PHP ABI records;
- validate signatures against typed Haxe code;
- generate repetitive Haxe facade modules;
- emit typed manifests for globals, classes, file segments, template segments, references, and native-value use;
- reject unsupported public shapes at compile time.

Macros must not perform broad textual rewrites of generated PHP. `Context.onAfterGenerate` may trigger or feed the linker, but the linker owns WordPress-facing file generation.

### Deterministic Linker

The linker owns WordPress-facing PHP files. It must:

- emit exact original paths;
- preserve file load order and include/require semantics;
- emit global functions and conditional declarations;
- emit public PHP shells for classes/interfaces where exact native identity matters;
- preserve top-level side effects, include return values, `ABSPATH` guards, template output, and caller-scope bridges;
- keep public PHP globals, superglobals, arrays, references, and callbacks native;
- delegate only bounded implementation payloads to Haxe-generated classes;
- produce provenance, ABI, file-segment, and generated-output manifests;
- be deterministic for identical inputs.

Generated shells are not a temporary hack. They are the accepted compatibility boundary for public WordPress PHP.

### Backend Fork

A Haxe PHP backend fork is allowed only after a minimized generic blocker proves that ordinary Haxe, target-specific APIs, macros, and the linker cannot preserve behavior safely or efficiently.

Before creating a backend task, all of these must be true:

1. a minimized fixture demonstrates the blocker;
2. the blocker is generic to procedural PHP ports, not a single WordPress call site;
3. a shell/linker solution would change behavior or impose unacceptable overhead;
4. the expected backend change has compiler-level tests;
5. the fork delta can be pinned, reviewed, and documented;
6. an ADR or ADR amendment accepts the maintenance cost.

The current F1-F7 evidence does not meet that threshold.

## Public PHP Policy

Public PHP boundaries must preserve the observable PHP contract first. That means:

- exact names and reflection-visible signatures;
- native PHP array/global/reference behavior;
- original file timing and side effects;
- caller-scope behavior for templates;
- native PHP callbacks for hooks;
- upstream-compatible warnings, falsey values, and return shapes where tested.

Haxe ownership increases behind these boundaries as evidence accumulates. Do not move a boundary inward merely because a typed Haxe design looks cleaner.

## Template and HHX Policy

HHX is allowed for Haxe-owned markup units and bounded generated fixtures. It is not parity evidence for existing mixed PHP/HTML WordPress files until the file-segment/template model has classified the original file's inputs, outputs, caller locals, globals, includes, output buffering, and return/exit behavior.

Existing mixed PHP/HTML files start as original-path shells with explicit renderer, context-bridge, or direct-script-emission mode.

## Smell Fixes

This architecture permits smell fixes, but not silent behavior changes.

Before parity, a smell fix must be marked as no-observable-change and cite oracle evidence. Intentional behavior changes require an ADR and must remain isolated from compatibility claims.

## Consequences

- WPHX-200/WPHX-205 should build the ABI extractor, manifest schema, and deterministic original-path linker instead of starting with a compiler fork.
- WPHX-209 remains conditional and should not be started unless a later minimized backend blocker appears.
- Generated WordPress-facing PHP is production architecture, not only test scaffolding.
- Haxe implementation code should avoid broad `Dynamic` at public boundaries except where PHP callbacks or native values require it.
- The PHP feasibility gate has enough evidence to proceed to ABI/linker work, while full WordPress parity remains gated by later upstream/differential tests.

## Enforcement

- Port slices must cite ABI, oracle, generated-output, or receipt evidence before claiming compatibility.
- Precommit and prepush hooks continue to enforce Haxe formatting and secret scanning.
- `npm run receipts:validate` must pass after closing tasks with receipts.
- Any future backend fork proposal must cite this ADR and include the minimized blocker fixture.

## Supersession

This ADR may be superseded if later evidence shows that the linker boundary cannot preserve a required public PHP contract or imposes unacceptable measured overhead. Supersession requires a new ADR, a minimized fixture, and updated receipts.
